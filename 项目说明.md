# API Search Agent 单页面 Web 项目技术文档

## 1. 项目目标

本项目要做一个面向 API 文档的智能检索与问答系统。用户可以持续上传 `.dts`、`.json` 等 API 文档文件，后端解析并增量更新统一索引和向量数据库；用户也可以用自然语言描述自己想要的接口，系统返回最匹配的 API 文档、关键信息、请求体、返回体，并提供入口查看完整详情。

当前仓库已经具备很好的数据底座：

| 资产 | 当前状态 |
|---|---|
| 原始 `.dts` 文件 | `API服务(openapi_apilist)_A374/`，约 1781 个 |
| 解析脚本 | `parse_dts.py`，可从 `.dts` 抽取统一 JSON |
| 统一索引 | `api_index.json`，1781 条 API |
| 汇总文档生成 | `gen_summary.py` |
| API 类型分布 | 标准 API 1485 个，自定义 API 296 个 |
| 解析质量 | 1776 条完整，5 条 partial |

第一版应基于现有 `parse_dts.py` 和 `api_index.json` 扩展，不要一开始重写解析逻辑。

## 2. 核心交互

### 2.1 API 实时更新

用户在页面上传一个或多个文件：

- `.dts`：金蝶云苍穹/星瀚导出的 API 元数据。
- `.json`：可支持两类格式：
  - 已解析后的 `api_index.json` 数组格式。
  - 标准 OpenAPI JSON，后续版本再支持完整适配。
- `.zip`：后续版本支持批量打包上传。

后端处理流程：

1. 接收上传文件并生成 `job_id`。
2. 保存原始文件到 `storage/uploads/{job_id}/`。
3. 根据扩展名选择 parser adapter。
4. 解析成统一 `ApiDoc` 结构。
5. 计算内容哈希 `content_hash`。
6. 与数据库中现有记录对比，识别新增、更新、删除或跳过。
7. 写入关系型数据库保存元数据和版本。
8. 生成检索文本与向量。
9. 批量 upsert 到向量数据库。
10. 更新任务状态，前端实时显示进度。

### 2.2 自然语言查找 API

用户输入类似：

- `我要查即时库存的接口`
- `销售订单查询接口，请给我请求体和返回体`
- `有没有按物料编码查询 BOM 的 API`
- `找一个可以批量审核采购订单的接口`

系统返回：

- 推荐 API 列表，包含匹配分数和命中原因。
- 首选 API 的完整关键信息。
- 请求方法、URL、所属云、应用、接口编码、说明。
- 请求参数树，标记必填、类型、层级、示例。
- 返回参数树，标记类型、层级、说明、示例。
- 可复制的请求体模板和返回体结构示例。
- “查看详情”入口，打开完整 API 文档面板。

## 3. 推荐技术栈

### 3.1 前端

| 模块 | 技术 | 说明 |
|---|---|---|
| 构建 | Vite + React + TypeScript | 单页应用，启动快，适合前后端分离 |
| UI | CSS Modules 或 Tailwind CSS | 第一版建议 CSS Modules，减少样式体系成本 |
| 请求 | 原生 `fetch` 封装 | 当前页面交互简单，避免过早引入复杂状态库 |
| 状态 | React hooks | 上传任务、搜索结果、详情面板状态都可本地管理 |
| 代码高亮 | `react-syntax-highlighter` 或轻量 JSON viewer | 展示请求体和返回体模板 |
| 图标 | `lucide-react` | 上传、搜索、详情、复制、刷新等按钮 |

页面不需要路由。所有功能集中在一个 SPA 首页里，通过面板、抽屉、标签页完成交互。

### 3.2 后端

| 模块        | 技术                                              | 说明                                  |
| --------- | ----------------------------------------------- | ----------------------------------- |
| Web 框架    | FastAPI                                         | 类型友好，天然生成 OpenAPI 文档，适合上传和流式接口      |
| 数据校验      | Pydantic                                        | 定义 `ApiDoc`、搜索响应、上传任务等结构            |
| 元数据存储     | SQLite 起步，PostgreSQL 生产                         | MVP 可直接 SQLite；生产建议 PostgreSQL      |
| ORM       | SQLModel 或 SQLAlchemy 2.x                       | 表结构清晰，便于后续迁移                        |
| 向量库       | Qdrant                                          | 支持 payload、过滤、批量 upsert、向量相似度搜索     |
| 后台任务      | MVP 用 FastAPI BackgroundTasks；生产用 Celery/RQ/Arq | 上传解析和 embedding 较慢，必须异步化            |
| Agent 编排  | 第一版手写 pipeline；复杂后使用 LangGraph                  | 先保证可控和可测，再引入 agent 编排框架             |
| Embedding | 可插拔 provider                                    | 支持云端 embedding 或本地 embedding，保持接口抽象 |

### 3.3 为什么建议 Qdrant

API 文档检索需要同时保存向量和结构化元数据，例如 `cloud`、`app`、`api_type`、`method`、`number`、`url`。Qdrant 的 point 模型由 vector 和 payload 组成，适合把每个 API 或 API chunk 做成检索点；查询时也能结合 payload filter 做限定检索。

## 6. 核心数据模型

### 6.1 ApiDoc

统一的 API 文档结构应兼容当前 `api_index.json`。

```python
class ApiParam(BaseModel):
    name: str
    type: str = ""
    desc: str = ""
    required: bool = False
    level: int = 1
    is_list: bool = False
    example: str = ""
    id: str | None = None
    parent_id: str | None = None


class ApiDoc(BaseModel):
    api_id: str
    file: str
    api_type: str
    partial: bool = False
    name: str
    number: str
    method_name: str = ""
    url: str
    http_method: str
    app: str = ""
    cloud: str = ""
    group: str = ""
    description: str = ""
    class_name: str = ""
    version: str = ""
    request_params: list[ApiParam] = []
    response_params: list[ApiParam] = []
    search_text: str
    content_hash: str
    source_type: Literal["dts", "json", "openapi"]
    created_at: datetime
    updated_at: datetime
```

`api_id` 建议使用稳定规则生成：

```text
api_id = sha1(source_namespace + "|" + url + "|" + number + "|" + name)
```

如果 URL 稳定且唯一，也可以把 URL 作为主身份的一部分，但不要只依赖文件名，因为上传更新后文件名可能变化。

### 6.2 ApiDocVersion

用于追踪实时更新历史。

```python
class ApiDocVersion(BaseModel):
    id: str
    api_id: str
    content_hash: str
    source_file: str
    source_job_id: str
    raw_snapshot_path: str | None
    parsed_snapshot_path: str | None
    change_type: Literal["created", "updated", "unchanged", "deleted"]
    created_at: datetime
```

### 6.3 UploadJob

```python
class UploadJob(BaseModel):
    job_id: str
    status: Literal["queued", "parsing", "indexing", "completed", "failed"]
    total_files: int
    parsed_files: int = 0
    total_apis: int = 0
    created_apis: int = 0
    updated_apis: int = 0
    skipped_apis: int = 0
    failed_files: int = 0
    error_message: str | None = None
    started_at: datetime | None = None
    finished_at: datetime | None = None
```

### 6.4 Vector Point Payload

向量库中的 payload 不保存完整大文档，只保存检索过滤和回查所需字段。

```json
{
  "api_id": "sha1:...",
  "chunk_id": "sha1:...",
  "chunk_type": "overview|request|response|examples|full",
  "name": "即时库存查询",
  "number": "AI_inv_realbalance",
  "url": "/v2/mzv0/im/im_inv_realbalance/AI_inv_realbalance",
  "http_method": "POST",
  "api_type": "标准API",
  "cloud": "供应链",
  "app": "库存管理",
  "group": "",
  "content_hash": "sha1:...",
  "version": "",
  "partial": false
}
```

## 7. 文档解析设计

### 7.1 Parser Adapter 接口

所有上传格式统一转成 `list[ApiDoc]`。

```python
class ParserAdapter(Protocol):
    supported_extensions: set[str]

    def parse_file(self, path: Path) -> list[ApiDoc]:
        ...
```

第一版实现：

| Adapter | 输入 | 输出 |
|---|---|---|
| `DtsParserAdapter` | 单个 `.dts` | 单个 `ApiDoc` |
| `ParsedJsonAdapter` | 当前 `api_index.json` 数组 | 多个 `ApiDoc` |
| `OpenApiJsonAdapter` | OpenAPI JSON | 多个 `ApiDoc`，可第二阶段开发 |

### 7.2 复用当前 parse_dts.py

当前脚本已经完成这些能力：

- 定位 `.dts` 中真正包含 `urlformat` 的 API 定义段。
- 抽取接口名、编码、URL、HTTP method、云、应用、分组、说明。
- 抽取请求参数 `bodyentryentity`。
- 抽取返回参数 `respentryentity`。
- 生成 `search_text`。
- 对截断文件做 salvage。

后端化时建议做两步：

1. 先把 `parse_dts.py` 复制或移动为 `backend/app/services/parsers/dts_parser.py`。
2. 保持函数名和测试样例稳定，只在输出阶段补齐 `api_id`、`content_hash`、`source_type`。

不要在第一版引入 LLM 来解析 `.dts`，解析应尽量确定性。

### 7.3 参数树构建

当前 `request_params` 和 `response_params` 保留了：

- `level`
- `_id`
- `_pid`

前端展示表格可以直接用 `level` 缩进。后续如果需要树形 JSON 模板，应在后端构建：

```python
def build_param_tree(params: list[ApiParam]) -> list[ParamNode]:
    # 优先使用 id / parent_id；缺失时按 level 栈回退。
    ...
```

## 8. 索引和向量化设计

### 8.1 检索文本分块

API 文档不是普通文章。用户多数是在找“哪个接口”，所以不能只做细粒度 chunk，否则会命中参数而丢失接口整体。建议每个 API 生成 3 到 5 个 chunk：

| chunk_type | 内容 | 用途 |
|---|---|---|
| `overview` | 名称、编码、URL、方法、云、应用、说明 | 找接口主意图 |
| `request` | 请求参数，含层级、必填、说明、示例 | 根据入参需求找接口 |
| `response` | 返回参数，含层级、说明、示例 | 根据返回字段找接口 |
| `full` | 当前 `search_text` 完整内容 | 兜底召回 |
| `examples` | 请求/返回模板，后续生成 | 示例型查询 |

第一版至少做 `overview` 和 `full`。参数很多的接口再拆 `request`、`response`。

### 8.2 content_hash

每个 API 的 hash 应基于规范化后的核心字段，而不是原始文件全文。

```python
hash_input = {
    "name": api.name,
    "number": api.number,
    "url": api.url,
    "http_method": api.http_method,
    "cloud": api.cloud,
    "app": api.app,
    "description": api.description,
    "request_params": api.request_params,
    "response_params": api.response_params,
}
content_hash = sha1(json.dumps(hash_input, ensure_ascii=False, sort_keys=True))
```

这样能避免文件导出顺序、无关字段变化造成无意义重建。

### 8.3 增量更新策略

上传任务中每条 API 处理逻辑：

```text
if api_id not exists:
  insert metadata
  insert ApiDocVersion(change_type="created")
  upsert vector chunks
elif content_hash changed:
  update metadata
  insert ApiDocVersion(change_type="updated")
  delete old chunks by api_id or upsert deterministic chunk_id
  upsert new vector chunks
else:
  insert ApiDocVersion(change_type="unchanged") 可选
  skip embedding
```

删除策略：

- 单文件上传不应该删除库中未出现在本次上传的 API。
- “全量重建”模式才允许标记缺失 API 为 deleted。
- 页面需要把上传模式显示清楚：`增量更新` 和 `全量重建`。

### 8.4 向量 collection

建议 collection 命名：

```text
api_docs_v1
```

如果更换 embedding 模型或维度，创建新 collection：

```text
api_docs_v2
```

不要在同一 collection 混用不同维度或不同语义空间的向量。

## 9. 检索策略

单纯向量检索对 API 场景不够稳定。应使用 hybrid retrieval：

### 9.1 召回层

并行执行：

1. 精确召回：
   - URL 完全匹配或局部匹配。
   - `number` 完全匹配。
   - `method_name` 完全匹配。
   - API 名称包含关键词。
2. 关键词召回：
   - SQLite FTS5 或 PostgreSQL full-text。
   - 查询中文时可先用简单 n-gram，后续再接分词。
3. 向量召回：
   - 使用用户 query embedding 搜 Qdrant。
   - 可按 `cloud`、`app`、`api_type` 过滤。
4. 同义词扩展：
   - `查询` ≈ `getList`、`query`、`batchQuery`
   - `新增` ≈ `add`、`batchAdd`
   - `修改` ≈ `update`、`batchUpdate`
   - `审核` ≈ `audit`、`batchAudit`
   - `反审核` ≈ `unaudit`、`batchUnaudit`
   - `撤销` ≈ `unsubmit`、`batchUnSubmit`

### 9.2 排序层

对候选 API 做融合排序：

```text
final_score =
  0.45 * vector_score
  + 0.25 * keyword_score
  + 0.20 * exact_score
  + 0.10 * field_match_score
```

`field_match_score` 可考虑：

- query 中出现了云/应用/业务对象。
- query 中出现了动作词，且 API 名称或 number 对应。
- query 中提到“批量”，API 名称或 number 包含 batch。
- query 中提到“列表”，API 名称或 number 包含 getList。

### 9.3 消歧策略

如果 top 1 和 top 2 分数差距小，例如：

```text
top1.score - top2.score < 0.08
```

前端应展示多个候选，并提示用户选择，而不是强行回答一个。

## 10. Agent 设计

严格来说，本项目是 agent 项目，但第一版不应该让 LLM 接管所有逻辑。建议把 agent 定义为“可观察、可测试的多步骤检索编排器”。

### 10.1 Agent 输入输出

输入：

```json
{
  "query": "我要查即时库存的接口，请给我请求体",
  "filters": {
    "cloud": "供应链",
    "api_type": "标准API"
  },
  "top_k": 5
}
```

输出：

```json
{
  "answer_type": "single|multiple|not_found",
  "summary": "推荐使用“即时库存查询”接口。",
  "selected_api_id": "sha1:...",
  "candidates": [
    {
      "api_id": "sha1:...",
      "name": "即时库存查询",
      "number": "AI_inv_realbalance",
      "url": "/v2/...",
      "score": 0.91,
      "reason": "名称命中“即时库存”，接口说明和查询动作匹配"
    }
  ],
  "doc": {
    "name": "...",
    "url": "...",
    "http_method": "POST",
    "request_params": [],
    "response_params": [],
    "request_template": {},
    "response_template": {}
  },
  "trace": {
    "rewritten_query": "...",
    "retrieval_methods": ["exact", "keyword", "vector"],
    "warnings": []
  }
}
```

### 10.2 Agent 节点

```text
User Query
  ↓
Query Normalizer
  - 清洗输入
  - 抽取动作词、业务对象、约束条件
  - 生成同义词
  ↓
Retrieval Planner
  - 判断是否需要 exact / keyword / vector / filter
  ↓
Candidate Retriever
  - SQL exact
  - FTS/BM25
  - Qdrant vector
  ↓
Reranker
  - 规则融合排序
  - 可选 LLM/rerank 模型重排
  ↓
Doc Builder
  - 从 DB 取完整 API 文档
  - 构造请求体/返回体模板
  ↓
Verifier
  - 确认输出字段均来自数据库
  - 没有 source 的内容不输出为事实
  ↓
Response
```

### 10.3 LLM 的边界

LLM 可以做：

- query 改写。
- 意图分类。
- 候选解释。
- 把参数说明整理成更自然的摘要。

LLM 不应该做：

- 编造 URL。
- 编造请求参数或返回参数。
- 修改必填状态。
- 生成不存在的接口。

所有结构化 API 文档字段必须来自数据库。`Verifier` 要校验输出中的 URL、number、参数名都存在于 `ApiDoc`。

### 10.4 第一版是否必须使用 LangGraph

不必须。建议路线：

1. MVP：用普通 Python service 实现固定 pipeline，方便调试。
2. 当需要多轮对话、人工确认、任务中断恢复、trace 可视化时，再迁移到 LangGraph。

LangGraph 适合做长流程、有状态、可恢复的 agent 编排；本项目第一阶段主要难点是数据解析、增量索引和检索质量，先把这些打牢。

## 12. 请求体和返回体模板生成

用户需要“请求体/返回体”，参数表还不够。后端应基于参数树生成模板。

### 12.1 生成规则

| 参数类型 | 模板值 |
|---|---|
| `String` | 示例值，否则 `""` |
| `Long` / `Integer` | 示例值，否则 `0` |
| `Decimal` | 示例值，否则 `0` |
| `Boolean` | 示例值，否则 `false` |
| `Date` / `DateTime` | 示例值，否则 `"2026-01-01"` |
| `Entries` / 对象 | `{}` |
| `is_list = true` | `[child_template]` |

### 12.2 必填过滤

前端提供切换：

- `完整模板`：展示所有字段。
- `仅必填字段`：只展示 required 字段和其必要父节点。

### 12.3 模板示例

```json
{
  "material_number": "",
  "org_number": "",
  "pageIndex": 1,
  "pageSize": 100
}
```

## 13. 前端页面设计

### 13.1 页面布局

单页工作台，不做 landing page。

```text
┌────────────────────────────────────────────────────────────┐
│ 顶部栏：API Search Agent | 索引状态 | 最后更新时间          │
├───────────────┬────────────────────────────────────────────┤
│ 左侧面板       │ 主工作区                                    │
│ - 上传文件     │ ┌────────────────────────────────────────┐ │
│ - 上传任务     │ │ 搜索框                                  │ │
│ - 过滤器       │ └────────────────────────────────────────┘ │
│ - 数据统计     │ 候选 API 列表                              │
│               │ 首选 API 摘要                              │
│               │ 请求体/返回体 tabs                          │
└───────────────┴────────────────────────────────────────────┘
                      右侧抽屉：完整 API 详情
```

### 13.2 组件职责

| 组件 | 职责 |
|---|---|
| `UploadPanel` | 文件拖拽、选择、上传模式、提交 |
| `JobTimeline` | 显示解析、索引、完成、失败 |
| `SearchPanel` | 自然语言输入、过滤器、搜索按钮 |
| `ResultList` | 展示候选 API、分数、命中理由 |
| `ApiSummary` | 首选 API 关键信息 |
| `ParamTable` | 请求参数/返回参数表格 |
| `JsonTemplate` | 请求体/返回体 JSON 模板，可复制 |
| `ApiDetailDrawer` | 完整文档详情，含版本和源文件 |
| `IndexStats` | 当前 API 数量、chunk 数、更新时间 |

### 13.3 关键交互状态

上传：

- idle：未选择文件。
- selected：已选择文件。
- uploading：正在上传。
- processing：后端解析/索引中。
- completed：显示新增、更新、跳过数量。
- failed：显示错误文件和错误原因。

搜索：

- empty：未输入。
- loading：检索中。
- single：有明确首选 API。
- multiple：多个候选接近，需要用户选择。
- not_found：没有足够匹配结果，提示换关键词或上传文档。
- error：后端错误。

详情：

- 点击候选卡片或“查看详情”打开右侧 drawer。
- 请求参数、返回参数、请求模板、返回模板用 tabs 切换。
- URL、number、JSON 模板提供复制按钮。

## 17. 生产注意事项

### 17.1 文件上传安全

- 限制扩展名。
- 限制大小。
- 原始文件保存到隔离目录。
- 不信任上传文件名，服务端生成安全路径。
- JSON 解析设置大小限制。

### 17.2 并发和一致性

- 同一时间只允许一个全量重建任务。
- 增量任务可以排队。
- Qdrant upsert 和 DB 更新要有可恢复策略。
- 任务失败时保留 job log。

### 17.3 可观测性

每次搜索记录：

- query
- filters
- top candidates
- selected api
- retrieval methods
- latency
- agent warnings

这些数据后续可用于优化同义词和 rerank。

### 17.4 防幻觉

- API URL、参数名、参数说明必须来自数据库。
- LLM 生成的摘要只作为补充。
- 前端应区分“原始文档字段”和“智能摘要”。
- 如果匹配不确定，返回候选列表，不强答。

## 18. MVP 最小闭环

如果只用 1 到 2 周做出可用版本，建议压缩为以下范围：

1. 后端 FastAPI。
2. SQLite 保存 API 元数据。
3. Qdrant 保存 `overview` 和 `full` 两种 chunk。
4. 复用当前 `.dts` 解析能力。
5. 支持上传 `.dts` 和当前 `api_index.json`。
6. 搜索使用 exact + vector，keyword 可以先简单实现。
7. 前端一个页面：上传、搜索、结果、详情。
8. 暂不做多轮对话，只做单轮 query。
9. 暂不用 LangGraph，只保留 `AgentService` 接口。

MVP 成功标准：

- 能全量导入当前 1781 个 API。
- 新上传文件能增量更新索引。
- 自然语言查询能返回正确 API 和请求/返回结构。
- 用户能从页面查看完整详情。

## 19. 后续增强

- 支持 OpenAPI JSON/YAML 完整导入。
- 支持 zip 批量上传。
- 支持多知识库或多环境隔离，例如 A374、2505。
- 支持多轮对话：“这个接口的必填字段有哪些？”、“换一个批量接口”。
- 支持用户反馈：正确/不正确，用于 rerank 优化。
- 支持混合检索中的 sparse vector 或 BM25。
- 支持 API 调用示例自动生成。
- 支持接口变更 diff。
- 支持权限：不同用户只能看授权 API 文档。

## 21. 当前 MVP 运行方式

npm run backend:dev
